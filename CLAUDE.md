# CLAUDE.md — The Bengal Reader

## Project overview

**The Bengal Reader** is a pure static civic journalism site covering the 2026 West Bengal Assembly Election. It tracks political corruption cases, MLA criminal records, asset declarations, electoral bonds, and BJP's post-election accountability. Non-partisan, data-driven, no framework.

**Live site:** https://voting-awareness-psi.vercel.app  
**Stack:** Vanilla HTML/CSS/JS + Node.js scripts + GitHub Actions + Vercel

## Development workflow

### Local setup

```bash
npm install          # installs htmlhint, stylelint, prettier (dev tools only)
npm run lint         # htmlhint on *.html + stylelint on *.css
npm run validate     # structure + data freshness checks (scripts/validate.mjs)
npm run generate-sitemap   # regenerate public/sitemap.xml
npm run generate-rss       # regenerate public/rss.xml
npm run build        # lint + generate-sitemap + validate (what CI runs)
```

No build step for pages — open any `public/*.html` directly in a browser, or run any static file server pointed at `public/`.

### Branches and CI/CD

- **`main`** — production branch; every push triggers a Vercel deploy
- **`claude/**`** — any branch matching this pattern is **automatically squash-merged into `main`** by `automerge-claude.yml` after every push, then Vercel auto-deploys within ~30 seconds
- CI (`ci.yml`) runs on push to `main` and `claude/**` branches and on PRs targeting `main`; it lints HTML/CSS, validates data, regenerates the sitemap, and checks for uncommitted changes

**When pushing code changes, always push to the `claude/**` branch** — the automerge workflow handles landing them on `main`.

### CI checks that will fail your PR

1. HTML missing `<!DOCTYPE html>`, `lang` attribute, `<meta charset>`, viewport, or `<title>`
2. Uncommitted changes left after `generate-sitemap` runs (sitemap is generated in CI and the diff must be zero)
3. Stylelint errors in `public/*.css`

## HTML page conventions

Every page must include all of the following or `validate.mjs` will warn/error:

```html
<!DOCTYPE html>
<html lang="en" data-theme="light">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <meta name="description" content="…">
  <meta property="og:title" content="…">
  <meta property="og:description" content="…">
  <meta property="og:image" content="/og-*.svg">
  <title>The Bengal Reader — …</title>
  <link rel="icon" type="image/svg+xml" href="/favicon.svg">
  <link rel="stylesheet" href="/shared.css">      <!-- always include -->
  <script defer src="/analytics.js"></script>     <!-- always include -->
  <script defer src="/shared.js"></script>        <!-- always include -->
  <!-- page-specific <style> block goes here -->
</head>
<body>
  <!-- Nav (copy from any existing page) -->
  <nav class="site-nav">…</nav>

  <main>…page content…</main>

  <!-- Page-specific <script> block at end of body -->
</body>
</html>
```

**CSS convention:** Page-specific styles live in a `<style>` block in `<head>`. `shared.css` provides nav, scroll bar, theme variables, and reduced-motion. Do not create separate per-page CSS files.

**JS convention:** Page-specific JS lives in a `<script>` block at end of `<body>`. `shared.js` provides theme toggle (`toggleTheme()`), language toggle (`setLang(lang)`), scroll progress, share utilities, and active nav highlighting.

### Theme system

CSS custom properties drive theming. Both light and dark are defined in every page:

```css
:root { --paper: #f4f1ea; --ink: #1a1714; … }
[data-theme="dark"] { --paper: #141210; --ink: #f0ece3; … }
```

`shared.js` persists the user's choice in `localStorage` under key `theme`.

### Language toggle (en/bn)

Elements with class `en-only` are shown in English mode; `bn-only` in Bengali mode. `shared.js` persists under key `lang`. Language buttons use `class="lang-btn"` and `data-lang="en|bn"`.

### Non-interactive elements with onclick

If you add `onclick` to a `div`, `span`, `td`, `li`, etc., you must also add `role="button"` (or appropriate role) and `tabindex="0"` — otherwise `validate.mjs` will warn.

## Data pipeline

### Automated (do not edit these files by hand in CI-managed fields)

| File | Updated by | Frequency |
|------|-----------|-----------|
| `data/news.json` | `scripts/update.mjs` | Every 6 hours |
| `data/meta.json` (`autoChecked`) | `scripts/update.mjs` | Every 6 hours |
| `data/pledges.json` | `scripts/update.mjs` | Every 6 hours |
| `data/cases.json` (auto entries) | `scripts/update.mjs` | Every 6 hours |
| `data/constituencies.json` | `scripts/scrape-results.mjs` | 3× daily |
| `data/mlas.json` | `scripts/scrape-mlas.mjs` | Daily at 02:00 UTC |

### Manually maintained

| File | Description |
|------|-------------|
| `data/cases.json` (human entries) | Timeline entries without `autoAdded` — curated from court records |
| `data/assets.json` | MLA asset declarations |
| `data/parties.json` | Party policy dimension scores |

`data/meta.json` `dataNote` field distinguishes auto-generated vs human-curated entries: `autoAdded` field is set on system-generated timeline entries.

### `cases.json` structure

Each case object has `id`, `title`, `summary`, `lastUpdated` (YYYY-MM month string), and a `timeline` array. Timeline entries contain `date` (e.g. `"2026 Apr"`), `event` (string), and optionally `autoAdded` (ISO date), `autoFrom` (source URL), `source` (string).

### `constituencies.json` structure

Top-level `{ seats: [...], districts: [...] }`. Each seat has `ac` (AC serial 1–294), `name`, `winner`, `party`, `margin`, `verified` (boolean), and optionally `votes`.

## Scripts reference

All scripts are Node.js ESM (`import`/`export`) and run with `node scripts/<name>.mjs`. No external npm dependencies — only Node built-ins.

| Script | Purpose | Key outputs |
|--------|---------|-------------|
| `update.mjs` | Fetches Google News RSS, ED, CBI press releases; updates timelines | `data/news.json`, `data/cases.json`, `data/meta.json`, `data/pledges.json` |
| `scrape-mlas.mjs` | Scrapes ECI affidavit portal for all 294 MLAs | `data/mlas.json` |
| `scrape-results.mjs` | Scrapes ECI result pages for constituency data | `data/constituencies.json` |
| `generate-sitemap.mjs` | Generates `sitemap.xml` from `vercel.json` rewrites | `public/sitemap.xml` |
| `generate-rss.mjs` | Generates `rss.xml` from `news.json` | `public/rss.xml` |
| `validate.mjs` | Checks HTML structure, data freshness, mlas/constituencies counts | exit code 1 on errors |
| `auto-update-system.mjs` | Post-scrape orchestration pipeline | orchestrates multiple scripts |

`generate-sitemap.mjs` reads `SITE_DOMAIN` env var (without `https://`) to set the domain; defaults to `voting-awareness.vercel.app`.

## What NOT to do

- Do not create separate CSS files per page — styles go in `<style>` blocks
- Do not introduce a framework (React, Vue, etc.) — the site is intentionally framework-free
- Do not add a server-side build pipeline — `vercel.json` + static files is the entire deployment model
- Do not add npm runtime dependencies — only devDependencies for linting/formatting
- Do not manually edit `sitemap.xml` or `rss.xml` — they are always regenerated by scripts
- Do not edit auto-generated fields in `news.json` or `meta.json` by hand
- Do not skip `<link rel="stylesheet" href="/shared.css">` or the shared JS/analytics includes on any page
- Do not push directly to `main` — push to `claude/**` and let automerge handle it

## Deployment

Vercel watches `main`. Any push (including the bot's auto-merge squash commits) triggers a deploy within ~30 seconds. The `outputDirectory` is `public/`. No build command is needed — Vercel serves static files directly.

To use a custom domain, set `SITE_DOMAIN` (without `https://`) in Vercel project environment variables. The sitemap and RSS scripts read this variable automatically.

## GitHub Actions summary

| Workflow | Trigger | What it does |
|----------|---------|--------------|
| `ci.yml` | push to `main`/`claude/**`, PR→main | Lint HTML+CSS, validate, generate sitemap, check no uncommitted changes |
| `weekly-update.yml` | Every 6h (cron) + manual | Runs `update.mjs`, commits data changes, alerts on failure via GitHub issue |
| `scrape-results.yml` | 3× daily (cron) + manual | Runs `scrape-results.mjs`, commits constituency data |
| `mla-scraper.yml` | Daily 02:00 UTC + manual | Runs `scrape-mlas.mjs`, commits MLA data |
| `automerge-claude.yml` | push to `claude/**` | Opens/updates PR → main, immediately squash-merges it |

Bot commits use `user.name = "Bengal Reader Bot"` and `user.email = "bot@bengal-reader.app"`. Failure in `weekly-update.yml` or `mla-scraper.yml` opens a GitHub issue (label: `auto-update-failure` or `scrape-failure`) only if no open issue with that label already exists.
